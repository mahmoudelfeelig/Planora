This section reports measurements from running the released code and scripts.

\subsection{Research Questions}
We evaluate three practical questions:
\begin{itemize}[leftmargin=*]
  \item \textbf{RQ1 (Feasibility reliability):} does the retry/fallback pipeline consistently return a feasible schedule?
  \item \textbf{RQ2 (Runtime trade-off):} how much faster is requested \texttt{greedy} room mode than requested \texttt{cp\_rooms} on larger scenarios?
  \item \textbf{RQ3 (Fallback behavior):} how often do strict-room requests end up requiring greedy fallback?
\end{itemize}

\subsection{Setup}
Runs were generated with \texttt{scripts/run\_experiments.py} on modes
\texttt{small\_demo}, \texttt{block\_profs}, \texttt{mixed\_large}, and \texttt{target\_case},
seeds \(\{1,2\}\), and requested room modes \(\{\texttt{cp\_rooms}, \texttt{greedy}\}\).
Per run we logged instance size, solve status, attempt history, total CP runtime, and base/LS penalties.

Configuration:
\begin{itemize}[leftmargin=*]
  \item strict first-attempt budget: \(15\) s;
  \item per-attempt time limit: \(45\) s;
  \item workers: \(4\);
  \item objective enabled on first attempt;
  \item reliability strategies:
  \begin{itemize}[leftmargin=*]
    \item objective-off retry in the same room mode if the first attempt is non-feasible;
    \item \texttt{cp\_rooms} \(\rightarrow\) \texttt{greedy} fallback if still non-feasible.
  \end{itemize}
\end{itemize}

The JSONL output was converted into LaTeX rows by \texttt{scripts/jsonl\_to\_latex\_table.py}.

\subsection{Results}
Table~\ref{tab:perf} summarizes 16 runs in total.

\paragraph{RQ1 (Feasibility reliability).}
All 16/16 runs were feasible with the configured retry/fallback policy (\(100\%\) feasibility in every scenario/mode pair in the table).

\paragraph{RQ2 (Runtime trade-off).}
On larger scenarios, requested \texttt{greedy} is consistently faster than requested \texttt{cp\_rooms}:
\begin{itemize}[leftmargin=*]
  \item \texttt{block\_profs}: \(70.46\) s vs \(38.18\) s (\(\approx 1.85\times\) speedup),
  \item \texttt{mixed\_large}: \(83.62\) s vs \(45.62\) s (\(\approx 1.83\times\)),
  \item \texttt{target\_case}: \(81.69\) s vs \(38.38\) s (\(\approx 2.13\times\)).
\end{itemize}
On \texttt{small\_demo}, both requested modes are similar (\(\approx 15\) s median).

\paragraph{RQ3 (Fallback behavior).}
Among 8 runs that requested \texttt{cp\_rooms}, only 2 finished in \texttt{cp\_rooms} (both \texttt{small\_demo}); the remaining 6 completed via \texttt{greedy} fallback.
This indicates strict room integration is often the runtime bottleneck on larger generated scenarios under short strict budgets.

\paragraph{Quality note.}
The main table focuses on feasibility and CP runtime.
Local search was not enabled in this batch (\texttt{ls\_iters}=0), so LS columns are intentionally shown as \texttt{-}.
