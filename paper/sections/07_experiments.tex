This section reports empirical results obtained by running the released code.
We recommend reporting both feasibility/performance and schedule quality.

\subsection{Generated Instances}
We evaluate on generated instances produced by the project's generator modes (e.g., \texttt{small\_demo}, \texttt{target\_case}).
For each mode and seed, we record:
\begin{itemize}[leftmargin=*]
  \item number of programs/groups/courses/activities/rooms/staff,
  \item solver mode (\texttt{cp\_rooms} vs \texttt{greedy}),
  \item CP-SAT time limit, number of workers, objective on/off,
  \item feasibility status and runtime,
  \item soft penalty before/after local search.
\end{itemize}

The runs in Table~\ref{tab:perf} were produced by \texttt{scripts/run\_experiments.py}
on modes \texttt{small\_demo}, \texttt{block\_profs}, \texttt{mixed\_large}, and \texttt{target\_case}
with seeds \(\{1,2\}\), requested room modes \(\{\texttt{cp\_rooms}, \texttt{greedy}\}\),
strict time budget \(15\) seconds, global per-attempt time budget \(45\) seconds, and \(4\) workers.

We enabled two reliability strategies:
\begin{itemize}[leftmargin=*]
  \item retry the same room mode with objective disabled when the first objective-enabled solve is not feasible;
  \item if the requested mode is \texttt{cp\_rooms} and still non-feasible, fall back to \texttt{greedy} with objective disabled.
\end{itemize}

The generated JSONL output was converted to LaTeX rows using \texttt{scripts/jsonl\_to\_latex\_table.py}.

Observed trends:
\begin{itemize}[leftmargin=*]
  \item \texttt{small\_demo}: 100\% feasible in both requested modes with low median runtime (\(\approx 15\)s).
  \item \texttt{block\_profs} and \texttt{mixed\_large}: both are now 100\% feasible in the tested seeds under the retry/fallback pipeline.
  \item \texttt{target\_case}: 100\% feasible under the same pipeline; requesting \texttt{greedy} remains substantially faster than requesting \texttt{cp\_rooms}.
  \item For requested \texttt{cp\_rooms} on the larger scenarios, final extraction often uses the \texttt{greedy} fallback mode after strict-room attempts consume the initial budget.
\end{itemize}

\subsection{Public Benchmarks (Optional)}
If using public benchmarks (e.g., International Timetabling Competition instances~\cite{itc2007_track3,mccollum2010itc}), we must define a mapping from benchmark input to this paper's data model and constraints.
Only benchmarks that can be mapped without altering the stated constraints should be reported.
For curriculum-based course timetabling datasets, reporting should follow the published data/validation conventions~\cite{bonutti2012benchmarking,bettinelli2015overview}.
